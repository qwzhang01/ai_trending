# 安全策略

## 支持的版本

我们致力于维护以下版本的安全更新：

| 版本 | 支持状态          |
|------|-------------------|
| 0.1.x | ✅ 安全更新支持    |
| < 0.1.0 | ❌ 不再支持       |

## 报告安全漏洞

我们非常重视安全问题。如果您发现任何安全漏洞，请不要在公开渠道（如 GitHub Issues）中报告。

### 报告流程

1. **发送安全报告**：请将详细的安全问题描述发送至 [security@example.com](mailto:security@example.com)
2. **包含信息**：
   - 漏洞的详细描述
   - 复现步骤和代码示例
   - 可能的影响范围
   - 建议的修复方案（可选）
3. **响应时间**：我们会在 48 小时内确认收到报告，并在 7 个工作日内提供初步评估

### 漏洞奖励

虽然我们目前没有官方的漏洞奖励计划，但对于有价值的报告，我们会：
- 在修复公告中致谢（如果您同意）
- 提供项目相关的特别感谢

## 安全最佳实践

### 环境变量安全

- 不要将 `.env` 文件提交到版本控制
- 使用强密码和 API 密钥
- 定期轮换敏感凭据

### API 密钥管理

```bash
# 正确做法：使用环境变量
export OPENAI_API_KEY="your-api-key"

# 错误做法：硬编码在代码中
api_key = "your-api-key"
```

### 依赖安全

我们使用以下工具确保依赖安全：
- `uv` 进行依赖管理
- 定期更新依赖版本
- 监控已知漏洞

## 安全更新

安全更新将通过以下方式发布：
- GitHub Releases 的安全公告
- 版本号遵循语义化版本控制
- 重大安全修复会发布补丁版本
